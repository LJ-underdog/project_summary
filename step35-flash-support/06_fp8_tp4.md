## 1. 背景

FP8 tp=2 跑通后（子任务 5），尝试 FP8 tp=4。首次运行即 crash：

```
ValueError: The output_size of gate's and up's weight = 320
is not divisible by weight quantization block_n = 128.
File: ATOM/atom/model_ops/moe.py, Fp8MoEMethod.create_weights
```

`moe_intermediate_size=1280`，tp=4 时 `inter_dim=320`，`320 % 128 = 64 ≠ 0`，触发 hard check。

**特别之处**：这次使用了 agent team（16 个 agent）完成调试，并且在执行之前先用 project-summary skill 生成了任务计划，由独立 reviewer agent 审查计划逻辑后再执行。

## 2. 调查过程

调查经历了 **三次迭代**，每次发现并修复一个问题，但仍有新问题出现。

### 迭代 1：create_weights ValueError fix + weight 分配改用 padded_inter

**Phase 0**：确认 crash 在 `create_weights` L1567，而非 forward。

**Phase 1**（并行调查）：
- A1：BF16 tp=4 已有 padding 参考（`UnquantizedFusedMoEMethod` L489-518，635e59e）
- A2：weight loader 中存在 MXFP4 partial load 机制（`_load_w13` L2256-2257），当时推断这个机制可以处理将 checkpoint `[E,640,H]` 载入 `[E,768,H]` 的情况
- B1：scale shape 不变（ceil(320/128)=ceil(384/128)=3，padding 不改变 scale 块数）
- C1：stage2 dispatch inter=384 对齐（384%128=0 ✓）
- D1：ValueError check 可改为 padding-aware（策略 C）

**实施**：weight 分配改用 `padded_inter=384`（`torch.zeros`），ValueError check 改为 padding-aware。

**结果**：load 不再 crash，但 inference 输出完全 gibberish（随机多语言 token 噪声）。

### 迭代 2：weight padding 移到 _process_block_quant（镜像 BF16）

**根因诊断**：A2 的结论不正确——MXFP4 partial load 机制针对的是 expert 维度的部分加载，不是 inter_dim 维度的 shape 不匹配（640 vs 768）。将 `[E,640,H]` 载入 `[E,768,H]` 时，数据实际未正确放置（全零或错误布局），导致 weight 实质上无效。

**正确方法**：BF16 tp=4 的做法是先按原始大小加载，然后在 `process_weights_after_loading` 中 zero-pad。FP8 应类比，在 `_process_block_quant` 中 normalize 之后、shuffle 之前 zero-pad。

**实施**：
- `create_weights` weight 分配恢复原始大小（`intermediate_size_per_partition=320`）
- `_process_block_quant` 加入 zero-padding（gate+zeros+up+zeros → inter_pad=384）

**结果**：仍然 gibberish，TTFT 正常（92ms），说明计算路径在跑，但数值全错。

### 迭代 3：发现真正根因 — scale TP sharding floor→ceil

**关键调试**：在 `_process_block_quant` 加入 debug print，观察 scale 值：

```
sc13[0, 0:6, 0] = [0.000208, 0.000203, 1.0, 0.000210, 0.000197, 1.0]
```

**发现**：scale[2] = scale[5] = **1.0**（`torch.ones` 初始值，从未从 checkpoint 加载！）

**根因追踪**：

```
full model gate scale：10 N-blocks（1280/128=10）
TP=4 sharding 时 _load_w13 L2314：
  load_shard_size = loaded_weight.shape[shard_dim] // self.tp_size
                  = 10 // 4 = 2  ← floor！应该是 ceil=3
每 rank 只加载 2 个 scale blocks，第 3 个（partial block，覆盖 gate [256:320]）被跳过
→ scale[2] 留在 torch.ones 初始值 1.0
```

**影响**：`dequant(fp8_weight) = fp8_val × scale = fp8_val × 1.0`，而真实 scale ≈ 0.0002，放大约 **5000×** → 完全错误的激活值 → gibberish。

**FP8 tp=2 不受影响**：inter=640 时 gate scale 有 10 块，10÷2=5 exact，无 partial block 问题。

## 3. 根因

**三个独立问题**（需同时修复）：

| # | 问题 | 文件位置 | 影响 |
|---|------|---------|------|
| 1 | `create_weights` ValueError check 不支持 padded inter | L1566-1578 | 模型加载直接 crash |
| 2 | `_process_block_quant` 无 inter_dim padding | L1709 | weight 未对齐 CK kernel 要求 |
| **3（根本）** | `_load_w13`/`_load_w2` scale TP sharding 用 floor | **L2314 / L2353** | scale partial block 未加载，数值放大 5000× |

---

## 4. 解决方案

### Fix 1：create_weights — padding-aware check

**文件**：`ATOM/atom/model_ops/moe.py`，`Fp8MoEMethod.create_weights`

```python
# 在 block_n/block_k 赋值后，check 之前加入：
padded_inter = (intermediate_size_per_partition + block_n - 1) // block_n * block_n

# ValueError check 改为：（padded_inter 总是满足，留作防御性检查）
if padded_inter % block_n != 0:
    raise ValueError(...)
```

### Fix 2：_process_block_quant — zero-pad weight（镜像 BF16）

**文件**：`ATOM/atom/model_ops/moe.py`，`Fp8MoEMethod._process_block_quant`

```python
# 在 _normalize_weights_and_scales 之后、shuffle_weights 之前：
inter_dim = layer.w2_weight.shape[-1]  # 320
inter_pad = ceil(inter_dim / block_n) * block_n  # 384

w13_new = torch.zeros(E, 2 * inter_pad, hidden, dtype=w13.dtype)
w13_new[:, :inter_dim, :] = w13[:, :inter_dim, :]          # gate 有效数据
w13_new[:, inter_pad:inter_pad+inter_dim, :] = w13[:, inter_dim:, :]  # up 有效数据

w2_new = torch.zeros(E, hidden, inter_pad, dtype=w2.dtype)
w2_new[:, :, :inter_dim] = w2
```

### Fix 3（根本）：_load_w13 / _load_w2 — floor → ceil

**文件**：`ATOM/atom/model_ops/moe.py`，L2314 和 L2353

```python
# 修复前（floor，导致 partial scale block 未加载）：
load_shard_size = loaded_weight.shape[shard_dim] // self.tp_size

# 修复后（ceil + narrow 边界保护）：
load_shard_size = (loaded_weight.shape[shard_dim] + self.tp_size - 1) // self.tp_size
start = load_shard_size * tp_rank
size = min(load_shard_size, loaded_weight.shape[shard_dim] - start)
loaded_weight = loaded_weight.narrow(shard_dim, start, size)
```

**回归安全**：
- tp=2 scale（10/2=5 exact）：ceil=floor=5，行为不变 ✓
- tp=1（无 sharding，tp_size=1）：load_shard_size = full size，不影响 ✓
- weight tensor 本身的 inter_dim sharding（1280/4=320，320 整除 128）：ceil=floor，不影响 ✓

## 5. 验证结果

```bash
# FP8 tp=4
rm -rf /root/.cache/atom/* && cd /tmp && CUDA_VISIBLE_DEVICES=0,1,2,3 \
  AITER_LOG_LEVEL=WARNING \
  python -m atom.examples.simple_inference \
  --model stepfun-ai/Step-3.5-Flash-FP8 --trust-remote-code \
  --tensor-parallel-size 4 --level 0 --temperature 0 --max-tokens 128 \
  --max-num-batched-tokens 4096 --max-num-seqs 256
```

**统一参数（公平对比）**：`--level 0 --temperature 0 --max-tokens 512 --max-num-batched-tokens 4096 --max-num-seqs 256 --gpu-memory-utilization 0.7`

| 配置 | GPU数 | gmu | TTFT | TPOT | 说明 |
|------|-------|-----|------|------|------|
| BF16 tp=2 | 2 | **0.9** | 92ms | 17.25ms | gmu=0.7 下 OOM（KV budget -12.86GB）|
| BF16 tp=4 | 4 | 0.7 | 88ms | 15.75ms | baseline |
| FP8 tp=2 | 2 | 0.7 | 85ms | **13.5ms** | TPOT 比 BF16 tp=4 快 14% |
| **FP8 tp=4** | **4** | **0.7** | **93ms** | **12.75ms** | **TPOT 比 BF16 tp=4 快 19%** |

**关键解读**：
- **TTFT（85-93ms）几乎相同**：4 prompts 共 76 tokens，prefill 极短，时间被 kernel launch + all-reduce 通信常数项主导，对权重精度不敏感。TTFT 不是区分 FP8/BF16 效率的合适指标（短 prompt 下）。
- **TPOT 才是关键**：decode 阶段 memory-bandwidth bound，FP8 权重仅 1 byte（BF16 2 bytes），理论带宽减半。实测 FP8 tp=4 比 BF16 tp=4 **TPOT 快 19%**（3ms 差异，约 7 倍标准差，统计显著）。
- **KV cache 优势**：同 gmu=0.7，FP8 tp=4 可用 KV 为 114GB（164k blocks），BF16 tp=4 仅 61GB（88k blocks）；BF16 tp=2 在 gmu=0.7 直接 OOM，FP8 tp=2 正常运行。
- **FP8 tp=2 性价比**：2 张 GPU，TPOT=13.5ms，仅比 FP8 tp=4（4 张 GPU）慢 5%，**tp=2 是更高性价比的选择**。

之前验证表（TTFT 为主、max-num-seqs 不统一）选错了指标，重测后 FP8 的优势清晰可见。

**10k token Prefill 性能（补充测试，2026-04-25）**

统一参数：`--enforce-eager --level 0 --max-tokens 5 --max-num-batched-tokens 16384 --max-num-seqs 4`

| 配置 | GPU数 | TTFT（10021 tokens） | Δ（GEMM 净时间） | 说明 |
|------|-------|---------------------|----------------|------|
| BF16 tp=2 | 2 | 466ms | 374ms | gmu=0.9 |
| BF16 tp=4 | 4 | 352ms | 264ms | gmu=0.7 |
| FP8 tp=2 | 2 | 444ms | 359ms | gmu=0.7，**比 BF16 快 4%** |
| **FP8 tp=4** | **4** | **319ms** | **226ms** | **gmu=0.7，比 BF16 快 14%** |

Δ = TTFT(10k) - TTFT(短 prompt)，代表真实 prefill GEMM 时间（扣除固定开销）。

**结论**：Prefill 阶段 FP8 同样有优势（tp=4 下快 14%），机制与 decode 不同——prefill 是 compute-bound（FP8 tensor core 算力），decode 是 bandwidth-bound（FP8 权重带宽）。TP 对两个阶段影响相反：prefill 用更多卡更快，decode 用更多卡因通信开销反而略慢。

## 7. 已知 Open Bug：tp=4 长序列 prefill 输出全 BOS

**发现日期**：2026-04-25
**状态**：⚠️ 未修复，需单独立项

### 现象

input tokens ≥ ~10k 时，tp=4 配置 prefill 完成后第一个 decode token 即为 BOS（token_id=0），且 512 个 output 全部为 BOS：

```
token_ids: [0, 0, 0, 0, ... 0]  # 512 个全部是 BOS
```

| 配置 | input=10021 tokens，max-tokens=512 | 结论 |
|------|-----------------------------------|------|
| BF16 tp=2 | 正常输出（"好的，用户给了一段"） | ✅ |
| FP8 tp=2 | 正常输出（"好的，用户给"） | ✅ |
| BF16 tp=4 | 512 × BOS（token_id=0） | ❌ |
| FP8 tp=4 | 512 × BOS（token_id=0） | ❌ |

日志：`/home/hanchang/project_fp8_tp4/logs/perf_compare_10k/bf16_tp4_512tok.log`，`fp8_tp4_512tok.log`

### 已确认事实（来自实验）

- **与 FP8/BF16 无关**：BF16 tp=4 同样失败，排除量化或 inter_dim padding 嫌疑
- **与输入长度相关**：日常 decode 测试（76 tokens）tp=4 输出完全正常；10021 tokens 时必然复现
- **prefill 完成后 logits 即错误**：第一个 decode token 就是 BOS，不是逐步退化的噪声累积

### 未调查的根因方向（假说，未验证）

Step-3.5-Flash 约 3/4 层使用 sliding window attention（window=512）。10k tokens 的 prefill 中，sliding window 的 mask 计算或 position encoding 在 tp=4 分布式路径下可能出错。历史上修复过 sliding window decode 的 off-by-one（aiter `7ebae9afb`），但 prefill 路径（`fmha_v3_varlen_fwd`）的 tp=4 行为未做深入验证。

### 与 SwigluStep BOS spam 的区别

SwigluStep 任务中的 BOS spam 是 bf16 噪声在 200+ decode steps 累积的结果（逐步退化）；此 bug 从第一个 token 开始就错误，是 prefill logits 的系统性问题，性质完全不同。

## 6. 教训

| 教训 | 说明 |
|------|------|
| **Scale 值检查法** | 对 FP8 量化模型，加载后立即检查 scale tensor 是否为 `torch.ones` 初始值（scale=1.0 与正常 ~0.0002 差异明显），可快速识别 scale 未加载问题 |
| 分层调试 gibberish | gibberish 可能来自：weight 未加载（全零）→ scale 错误（放大 5000×）→ 计算路径错误。要逐层确认，不要一次性归因 |
| Monkey-patch 在 spawn 子进程中失效 | ATOM 用 spawn 模式启动 EngineCore，monkey-patch 不会被子进程继承，必须直接修改源码加 debug print |
| Floor vs ceil 在 TP sharding 中的不对称 | weight 维度（1280/tp=320）和 scale 块数（1280/128/tp=2.5）的 TP 整除性不同，需要分别处理 |
| 三次迭代是正常的 | 每次迭代暴露下一层 bug：第 1 次 crash，第 2 次 weight 错，第 3 次 scale 错。不要期望一次性修复所有问题 |
